# MARK1（一人開発）Transformer学習 + 叩き台構築プラン

## 0. このプランの前提（今回の線引き）
- 主目的は **「本番品質」よりも、Transformer技術理解と叩き台の早期構築**。
- 開発者は **あなた一人**。
- AIコーディング支援を最大活用し、最短で動くものを作る。
- まずは **小規模・低コスト・短期間（最短7日、最大14日）** で完了させる。
- **重要方針**: PyTorchなどの高機能DLライブラリ依存を極力下げ、
  データ変形が追える実装を優先する。

## 1. Mark1のゴール再定義（最小）
- ゴール: 「TransformerベースのAIが動く」ことを自分の手で理解しながら再現する。
- 完了条件（最低ライン）
  1. 最小Transformerの前向き計算を自作コードで追える
  2. 最低1つのユースケース（例: Q&A）に回答できる
  3. モデルの仕組みを説明できる（Tokenizer / Attention / データ形状変化）
  4. 改善ポイントを次ステップに言語化できる

## 2. 今回「やること / やらないこと」

### やること（Must）
- まずは **NumPy中心** で最小Transformer（推論中心）を実装
- 各処理で shape をログ出力し、テンソル変形を可視化
- 可能なら最小限の学習（小さな次トークン予測）まで実施
- CLIで動作確認（UIは後回し）
- 評価は最小限（20〜50問の手動評価 + ログ確認）

### やらないこと（Won't for Mark1）
- PyTorch前提の複雑な学習パイプライン
- 大規模モデルの本格ファインチューニング
- RLHF/DPOの導入
- 複雑なマイクロサービス分割
- 厳密な本番運用（24/7監視、冗長化、SLO設計）

## 3. 学習優先の技術スコープ（理解テーマ）
優先順で理解する:
1. Tokenizer と token の概念
2. Embeddingと位置埋め込み
3. Self-Attention（Q/K/V、スコア計算、因果マスク）
4. Multi-Head結合と線形射影
5. FFN、残差接続、LayerNorm
6. 生成時のデコード（greedy / temperature付きサンプリング）

## 4. 推奨ミニマム構成（PyTorch極小依存）
- 言語: Python
- 数値演算: NumPy（必須）
- 実装方針: まずは自作Transformerブロック
- 補助ライブラリ: 可能なら標準ライブラリ中心（`json`, `argparse`, `time`）
- ログ: JSONLで入出力・shape・推論時間を記録
- CLI: `python main.py --prompt "..."`

> 高レベルライブラリで隠蔽される処理を減らし、「何がどの形で流れるか」を理解する。

## 5. 実装アーキテクチャ（最小）
- `tokenizer.py`: 文字または簡易サブワード分割
- `model.py`: Embedding / Attention / FFN / LayerNorm / 出力層
- `decode.py`: greedy と sampling
- `train_min.py`: 任意（超小規模データでの学習ループ）
- `main.py`: CLI入口
- `logs/*.jsonl`: shapeと出力結果ログ

## 9. 失敗しやすいポイントと回避策（1人開発向け）
- 実装を欲張って巨大化する
  - 回避: 1ブロック・小語彙・短文入力で開始。
- 数式理解と実装が分離する
  - 回避: 式ごとにshapeと中間値をログ出力。
- ライブラリ依存でブラックボックス化する
  - 回避: まずNumPy自作、後で必要最小限に置換。
- 評価が曖昧になる
  - 回避: 最初に質問20個を固定して比較。
